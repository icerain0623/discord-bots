# 1文リレー機能 設計書

## 概要

サーバーイベント用の1文リレー機能。各参加者が前の一文だけを見て次の文を追加し、最終的に完成した文章を投稿する。匿名で全文公開した後、任意のタイミングでネタバレ（執筆者一覧）を公開できる。

## コマンド一覧

すべてサーバー管理者（ManageGuild）のみ実行可能。サーバーにつき同時に1リレーのみ。

| コマンド | オプション | 説明 |
|---------|-----------|------|
| `/relay start` | `topic`(必須), `first_sentence`(必須) | お題と最初の一文を設定してリレー開始 |
| `/relay status` | なし | 全文を番号付きで表示（エフェメラル） |
| `/relay delete` | `number`(必須, Integer) | 指定番号の文を削除 |
| `/relay end` | `channel`(必須, Channel) | 全文（匿名）を指定チャンネルに投稿、ボタン無効化 |
| `/relay reveal` | `channel`(必須, Channel) | ネタバレを指定チャンネルに投稿、データ削除 |
| `/relay cancel` | なし | リレーを中止してデータ削除（投稿なし） |

## 参加フロー

1. 管理者が `/relay start` 実行
2. チャンネルにお題 + 「一文を追加」ボタンを含むメッセージが投稿される
3. ユーザーがボタンを押す
   - 直前の投稿者と同じ場合はエフェメラルでエラー表示
   - 前の一文をエフェメラルメッセージで表示しつつ、モーダルを開く
   - モーダルには入力欄のみ（Discordモーダルのlabelは45文字制限のため、前の一文はモーダル外で表示）
4. 送信後、KVに保存
5. チャンネルのリレーメッセージが更新される（現在N文目の進捗表示）
6. 管理者が `/relay end #channel` で全文を匿名投稿（ボタン無効化）
7. 管理者が `/relay reveal #channel` でネタバレ投稿、データ削除

## データ構造

### KV

ストレージ: `SESSION_KV`（既存のcontact機能と同じ名前空間を共有、キープレフィックスで区別）

キー: `relay-active:{guildId}` （TTL 30日、安全策。通常はreveal/cancelで明示的に削除）

```json
{
  "topic": "お題テキスト",
  "channelId": "111111111",
  "messageId": "222222222",
  "sentences": [
    {
      "text": "最初の一文",
      "userId": "123456789",
      "displayName": "管理者名"
    },
    {
      "text": "2番目の文",
      "userId": "987654321",
      "displayName": "ユーザー名"
    }
  ],
  "startedBy": "123456789",
  "startedAt": "2026-03-23T12:00:00.000Z"
}
```

- `channelId` / `messageId`: リレーパネルメッセージの参照（進捗更新・ボタン無効化に使用）
- `displayName`: 書き込み時点の表示名を保存（reveal時のAPI呼び出しを回避）

## コマンド詳細

### `/relay start`

1. `relay-active:{guildId}` が存在するか確認。存在すればエラー（既にリレー進行中）
2. チャンネルに以下のメッセージを投稿（`discordApi.js` の `postMessage` を使用）：
   - お題の表示
   - 「一文を追加」ボタン（customId: `relay_add`）
   - 進捗表示：「現在 1 文目」
3. 投稿されたメッセージの `channelId` と `messageId` を含めてKVにデータ保存
4. エフェメラルで「リレーを開始しました」と応答
5. deferred パターン使用（`type: 5` + `waitUntil`）：メッセージ投稿後にKV保存が必要なため

### 「一文を追加」ボタン（`relay_add`）

1. KVからリレーデータを取得。存在しなければエフェメラルでエラー
2. sentences配列の最後のuserIdと押したユーザーのIDを比較。同じならエフェメラルでエラー
3. モーダルを表示（customId: `relay_modal`, type: 9）：
   - テキスト入力欄（Short、必須、最大200文字）
   - ラベルは「あなたの一文を入力してください」等の固定テキスト
   - **注意**: Discordではモーダルとエフェメラルメッセージを同一インタラクションで同時に返せない。モーダルを優先し、前の一文はモーダルのtitle（最大45文字）に収まる範囲で表示する。45文字を超える場合は末尾を「…」で切り詰める

### モーダル送信（`relay_modal`）

1. KVからリレーデータを再取得（race condition対策）
2. 再度連続投稿チェック。失敗時はエフェメラルでエラー
3. sentences配列に新しい文を追加（text, userId, displayName）
4. KVに保存
5. エフェメラルで「追加しました！（N文目）」と応答
6. `waitUntil` でチャンネルのリレーメッセージを編集（進捗表示を更新）

### `/relay status`

1. KVからリレーデータを取得。存在しなければ「リレーは開催されていません」
2. エフェメラルで以下を表示：
   ```
   お題：〇〇（全N文）
   1. 一文目 — @ユーザーA
   2. 二文目 — @ユーザーB
   3. 三文目 — @ユーザーA
   ```

### `/relay delete`

1. KVからリレーデータを取得。存在しなければエラー
2. 指定番号のバリデーション（1以上、sentences.length以下）
3. sentences配列から該当要素を削除
4. sentencesが空になった場合もそのまま許容（次の参加者には「前の一文なし」と表示）
5. KVに保存
6. エフェメラルで「N番目の文を削除しました（残りM文）」と応答

### `/relay end`

1. KVからリレーデータを取得。存在しなければエラー
2. deferred パターン使用（`type: 5` + `waitUntil`）
3. 指定チャンネルに全文を投稿（2000文字超の場合は複数メッセージに分割）：
   ```
   **お題：〇〇**

   一文目。二文目。三文目。...
   ```
4. リレーパネルメッセージを編集：ボタンを削除し「リレー終了」表示に変更
5. KVデータは削除しない（revealのために保持）
6. フォローアップで「投稿しました」と応答

### `/relay reveal`

1. KVからリレーデータを取得。存在しなければエラー
2. deferred パターン使用（`type: 5` + `waitUntil`）
3. 指定チャンネルにネタバレを投稿（2000文字超の場合は複数メッセージに分割）：
   ```
   **お題：〇〇 — 執筆者一覧**

   1. 一文目 — @ユーザーA
   2. 二文目 — @ユーザーB
   3. 三文目 — @ユーザーA
   ```
4. KVデータを削除
5. フォローアップで「ネタバレを投稿しました。リレーデータを削除しました」と応答

### `/relay cancel`

1. KVからリレーデータを取得。存在しなければエラー
2. リレーパネルメッセージを編集：ボタンを削除し「リレー中止」表示に変更
3. KVデータを削除
4. エフェメラルで「リレーを中止しました」と応答

## エラーハンドリング

- リレー未開催時のend/reveal/delete/status/cancel → エフェメラルでエラー
- リレー開催中のstart → エフェメラルでエラー
- 連続投稿 → エフェメラルでエラー
- delete番号が範囲外 → エフェメラルでエラー
- KV読み書きのrace condition → モーダル送信時にKVを再取得して検証（KVの結果整合性の制約上、完全な排他は不可。イベント用途では許容範囲）
- チャンネルへの投稿失敗 → APIレスポンスを確認し、エラー時はフォローアップで通知
