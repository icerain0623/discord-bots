# Matchup: 交流マッチング機能 設計書

> **ステータス: テスト機能（experimental）** — 本機能は試験的な導入であり、フィードバックを元に仕様変更・削除の可能性がある。

## 概要

サーバーメンバー同士の交流を促進するマッチング機能。管理者が募集を開始し、参加者がトピックを選んで登録、管理者がマッチングを実行するとランダムにグループ分けされ、専用プライベートチャンネルで交流する。

## 利用シーン

既存メンバー同士で定期的に交流を促進するイベント（例：月1回のシャッフルマッチ）。

## コマンド構造

`/matchup` コマンドに以下のサブコマンド/サブコマンドグループを持たせる。全て `Manage Guild` 権限が必要。

| サブコマンド | 説明 |
|---|---|
| `/matchup start` | 募集メッセージを投稿（参加ボタン付き） |
| `/matchup run` | 参加者をマッチング＆プライベートチャンネル作成 |
| `/matchup terminate` | チャンネル一括削除・イベント終了（募集中でもキャンセル可能） |
| `/matchup topics add` | トピック追加（サブコマンドグループ） |
| `/matchup topics remove` | トピック削除（サブコマンドグループ） |
| `/matchup topics list` | トピック一覧表示（サブコマンドグループ） |

> **注:** `topics` はサブコマンドグループ。コマンド登録時に `addSubcommandGroup` を使用し、`interaction.data.options` のネストが1段深くなる。

### コマンドオプション

**`/matchup start`**
- `group_size` (必須, integer): グループサイズ（2〜4）
- `category` (任意, channel): チャンネル作成先カテゴリ。未指定なら自動作成

**`/matchup run`** — オプションなし

**`/matchup terminate`** — オプションなし

**`/matchup topics add`**
- `name` (必須, string): 追加するトピック名

**`/matchup topics remove`**
- `name` (必須, string): 削除するトピック名

**`/matchup topics list`** — オプションなし

## 参加フロー

### 1. 募集開始

管理者が `/matchup start group_size:2` を実行すると、募集メッセージ（Embed）が投稿される：

```
🎲 交流マッチング募集中！
グループサイズ: 2人
現在の参加者: 0人

[参加する ✋]
```

### 2. 参加登録（2段階インタラクション）

> **Discord API制限:** モーダル内にセレクトメニューは配置できない。そのため2段階で処理する。

**ステップ1:** ユーザーがボタンをクリックすると、エフェメラルメッセージで定義済みトピックのセレクトメニュー（複数選択可）を表示。

**ステップ2:** トピック選択後、「自由入力もありますか？」ボタンを表示。
- 「はい」→ モーダルが開き、フリーテキストで追加トピックを入力
- 「スキップ」→ そのまま登録完了

登録完了後、エフェメラルメッセージで確認：
```
✅ 参加登録しました！選択したトピック: ゲーム, 音楽, 「猫の話」
```

募集メッセージの参加者数はリアルタイム更新される。

### 3. 参加取消

同じボタンを再度クリックした場合：
- 「既に参加しています。取り消しますか？」とエフェメラルで確認ボタンを表示
- 取消後にトピックを変えて再参加も可能

## マッチング＆チャンネル作成

### 前提条件

- 最低2人の参加者が必要。不足している場合はエラーメッセージを返す。
- 既にマッチング済みの場合は `run` を拒否する（二重実行防止）。

### マッチングロジック

- 参加者をランダムにシャッフル
- 指定された `group_size` でグループに分割
- 端数は最後のグループに吸収（例：7人で2人ペアの場合 → 2, 2, 3）

### 非同期処理（Deferred Response）

`/matchup run` と `/matchup terminate` はチャンネル作成・削除等のDiscord API呼び出しが多いため、以下の手順で処理する：

1. Deferred message（`type: 5`）を即座に返す
2. `ctx.waitUntil()` でバックグラウンド処理を継続
3. 処理完了後、webhook で follow-up メッセージを送信

> **注:** 現在の `worker.js` の fetch ハンドラは `ctx`（ExecutionContext）パラメータを受け取っていないため、第3引数として追加が必要。

### チャンネル作成

専用カテゴリ内に各グループ用プライベートチャンネルを作成：
- チャンネル名: `matchup-1`, `matchup-2`, ...
- 権限: グループメンバー＋botのみ閲覧可能（`@everyone` は `VIEW_CHANNEL` を deny、各メンバーと bot の application ID に `VIEW_CHANNEL` を allow）

> **注:** Discordカテゴリは最大50チャンネル。`group_size:2` の場合、最大100人が実質上限。

### 初回メッセージ

botがチャンネルに挨拶メッセージを投稿：

```
🎲 マッチングされました！
メンバー: @Alice @Bob

💬 共通の話題: ゲーム, 音楽
📝 Aliceさんのトピック: ゲーム, 音楽, 「猫の話」
📝 Bobさんのトピック: 音楽, 映画

楽しく交流してください！
```

共通トピックがない場合は共通の話題の行を省略し、それぞれのトピックだけ表示する。

### 募集メッセージ更新

マッチング完了後、募集メッセージを更新：
```
🎲 マッチング完了！
4グループが作成されました。
```

## 終了処理

`/matchup terminate` は募集中・マッチング済みどちらの状態でも実行可能：

- **募集中の場合:** 募集をキャンセル。チャンネル削除は不要。
- **マッチング済みの場合:** 作成された全チャンネルを一括削除。

共通処理：
- 募集メッセージを更新：「🎲 このマッチングイベントは終了しました。」
- KVからイベントデータをクリーンアップ

### 将来的な拡張

- **チャットログエクスポート:** terminate 前にチャットログをエクスポートし、txt等のビューワーで閲覧可能にする
- **マッチング履歴:** 過去のマッチング結果を記録し、同じ組み合わせを避けるロジック
- **フィードバック機能:** 交流後に参加者が感想・評価を送信できる仕組み
- **定期自動実行:** cronトリガーやスケジュール機能と連携した自動募集・マッチング
- **Discordスケジュールイベント連携:** Gateway接続が可能になった場合、イベント機能との統合

## データ設計（KV）

専用の `MATCHUP_KV` 名前空間を `wrangler.toml` に追加する。既存の `SESSION_KV`（セッション用、30分TTL）とは分離する。

全てのキーにギルドIDプレフィックスを付与し、複数サーバー対応とする。

### `matchup-topics:{guildId}`
サーバーのトピック一覧（永続、TTLなし）。

```json
["ゲーム", "音楽", "映画", "アニメ", "料理"]
```

上限25個（Discordセレクトメニューの選択肢上限に合わせる）。

### `matchup-active:{guildId}`
現在アクティブなイベント情報（イベント中のみ存在）。

```json
{
  "status": "recruiting",
  "messageId": "123456",
  "channelId": "789012",
  "groupSize": 2,
  "categoryId": "345678",
  "participants": [
    { "userId": "111", "topics": ["ゲーム", "音楽"], "freeTopics": ["猫の話"] },
    { "userId": "222", "topics": ["音楽", "映画"], "freeTopics": [] }
  ],
  "createdChannels": ["ch_id_1", "ch_id_2", "ch_id_3"]
}
```

- `status`: `"recruiting"` | `"matched"` — 明示的な状態管理
- `participants`: 参加者一覧。小規模コミュニティ向けのため単一KV値で管理（KV値上限25MiB）

> **既知の制約:** KVのeventual consistencyにより、ほぼ同時の参加登録でread-modify-write競合が起こる可能性がある。小規模コミュニティでは実用上問題にならない想定。

### 制約
- アクティブなイベントは同時に1つのみ。`start` 時にアクティブがあればエラー。

## 状態遷移

```
なし → [start] → 募集中(recruiting) → [run] → マッチング済(matched) → [terminate] → なし
                         ↓
                    [terminate]
                         ↓
                        なし
```

各コマンドは `matchup-active` の `status` フィールドを検証し、不正な遷移はエラーメッセージを返す。

| 現在の状態 | start | run | terminate |
|---|---|---|---|
| なし | OK | エラー | エラー |
| recruiting | エラー | OK（2人以上） | OK（キャンセル） |
| matched | エラー | エラー | OK（削除） |

## トピック管理

- トピックはサーバーに1セット（グローバル）。イベントごとには分けない。
- `add`: 重複チェックあり。上限25個。
- `remove`: 存在しない場合はエラー。
- `list`: 現在のトピック一覧をEmbed表示。

## 技術的考慮事項

- 既存のCloudflare Workers + KVアーキテクチャに準拠
- `MATCHUP_KV` 名前空間を `wrangler.toml` に新規追加
- `worker.js` の fetch ハンドラに第3引数 `ctx`（ExecutionContext）を追加し、`ctx.waitUntil()` で非同期処理を実行
- インタラクション処理は既存の `worker.js` のルーティングに追加
- ボタン・セレクトメニュー・モーダル処理は既存の `interactions/` パターンに従う
- Discord API呼び出し（チャンネル作成・削除・権限設定・メッセージ編集）は `utils/discordApi.js` を拡張（現在はメッセージ取得のみのため、大幅な追加となる）
- bot の application ID は `interaction.application_id` から取得し、チャンネル権限設定に使用
